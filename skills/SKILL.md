# Frontend Design & UI/UX Skills

Combined design intelligence for Coordit — merges ui-ux-pro-max, frontend-design aesthetics, and project-specific design system rules.

---

## 1. Design Thinking (Before Coding)

Before writing a single line of code, commit to a clear aesthetic direction:

- **Purpose**: What problem does this interface solve? Who uses it?
- **Tone**: Pick a direction and execute it fully — brutally minimal, maximalist, editorial, luxury/refined, playful, retro-futuristic, industrial, etc.
- **Differentiation**: What's the one thing someone will remember about this screen?

**Rule**: Bold maximalism and refined minimalism both work. The key is intentionality, not intensity. Execute the chosen direction with precision.

---

## 2. Avoid Generic AI Aesthetic

These patterns make UI look like AI-generated slop. Avoid all of them:

| Avoid | Use Instead |
|-------|------------|
| Inter, Roboto, Arial, system-ui as default | Distinctive display + refined body font pairing |
| Purple gradients on white backgrounds | Context-specific palettes from color token system |
| Predictable card-grid-button layouts | Asymmetry, overlap, diagonal flow, grid-breaking elements |
| Evenly distributed, timid color palettes | Dominant base color + sharp accent |
| Shadow + rounded corner on every element | Deliberate elevation scale with purpose |
| Emojis as icons | SVG icon set (Lucide, Heroicons) — consistent stroke/style |
| Random hex values in components | Semantic color tokens only |
| Space Grotesk across every project | Vary per context — never converge on the same font |

**Mental check before delivery**: If this could have been generated by a first-pass AI prompt with no context — redesign it.

## 3. Typography Scale

Use a real type system. No random font sizes.

### Scale (rem / px reference)

| Token | Size | Usage |
|-------|------|-------|
| `text-xs` | 12px / 0.75rem | Captions, badges, metadata |
| `text-sm` | 14px / 0.875rem | Secondary labels, helper text |
| `text-base` | 16px / 1rem | Body text (minimum for mobile) |
| `text-lg` | 18px / 1.125rem | Lead text, card subtitles |
| `text-xl` | 20px / 1.25rem | Section labels, callouts |
| `text-2xl` | 24px / 1.5rem | Card headings, modal titles |
| `text-3xl` | 30px / 1.875rem | Page section headings |
| `text-4xl` | 36px / 2.25rem | Hero subheadings |
| `text-5xl` | 48px / 3rem | Hero headlines |
| `text-6xl` | 60px / 3.75rem | Display / marketing |

### Rules

- Body text minimum: **16px** (prevents iOS auto-zoom)
- Line height body: **1.5–1.75**
- Line length: **35–60 chars mobile / 60–75 chars desktop**
- Font weight hierarchy: Bold headings (600–700), Regular body (400), Medium labels (500)
- Use **tabular figures** for numbers in tables, prices, timers
- Never use `letter-spacing: tight` on body text
- Truncate with ellipsis + full text via tooltip when truncation is required

### Font Pairing Approach

Pair a distinctive display font with a refined body font. Examples:

- Editorial: `Playfair Display` + `Source Serif 4`
- Modern tool: `DM Mono` + `DM Sans`
- Luxury: `Cormorant Garamond` + `Jost`
- Playful: `Syne` + `Nunito`
- Engineering: `JetBrains Mono` + `IBM Plex Sans`

## 4. Spacing System (8px Base Grid)

All spacing values must be multiples of 4px or 8px. No arbitrary values.

### Spacing Tokens

| Token | Value | Usage |
|-------|-------|-------|
| `space-1` | 4px | Icon padding, micro gaps |
| `space-2` | 8px | Inline element gap |
| `space-3` | 12px | Tight component padding |
| `space-4` | 16px | Default component padding |
| `space-5` | 20px | Card internal padding |
| `space-6` | 24px | Section sub-spacing |
| `space-8` | 32px | Component-to-component gap |
| `space-10` | 40px | Section top/bottom padding |
| `space-12` | 48px | Large section separation |
| `space-16` | 64px | Page-level section breaks |
| `space-20` | 80px | Hero / landing sections |
| `space-24` | 96px | Maximum section separation |

### Rules

- Use spacing tokens — never `margin: 13px` or `padding: 7px`
- 8dp spacing rhythm maintained at: component level, section level, and page level
- Vertical rhythm tiers: `16 / 24 / 32 / 48` mapped to hierarchy levels
- Gutters on mobile: `16px`, tablet: `24px`, desktop: `32px`
- Touch targets: minimum `44×44px` (iOS) / `48×48dp` (Android)

## 5. Color Tokens

No raw hex codes in components. Define semantic tokens, reference only tokens.

### Token Structure

```css
/* Base palette — brand-specific */
--color-primary-50:  /* lightest tint */
--color-primary-100:
--color-primary-200:
--color-primary-300:
--color-primary-400:
--color-primary-500: /* base brand color */
--color-primary-600:
--color-primary-700:
--color-primary-800:
--color-primary-900: /* darkest shade */

--color-neutral-0:   /* white */
--color-neutral-50:
--color-neutral-100:
--color-neutral-200:
--color-neutral-300:
--color-neutral-400:
--color-neutral-500:
--color-neutral-600:
--color-neutral-700:
--color-neutral-800:
--color-neutral-900: /* near-black */

--color-accent-500:  /* single accent */

/* Semantic tokens — reference base palette */
--color-surface:          /* page/card background */
--color-surface-elevated: /* modal, dropdown */
--color-border:           /* dividers */
--color-border-strong:    /* focused inputs */

--color-text-primary:     /* headings, primary content */
--color-text-secondary:   /* captions, helper text */
--color-text-disabled:    /* disabled state */
--color-text-inverse:     /* text on dark bg */

--color-interactive:      /* links, buttons */
--color-interactive-hover:
--color-interactive-active:
--color-interactive-disabled:

--color-error:            /* form errors */
--color-error-surface:    /* error backgrounds */
--color-success:          /* confirmations */
--color-success-surface:
--color-warning:          /* caution states */
--color-warning-surface:
```

### Rules

- All component styles reference semantic tokens, never base palette tokens directly
- Dark mode: map semantic tokens to different base values — do not invert
- Error/success colors must meet **4.5:1 contrast** against their surface pair
- Never convey information by color alone — add icon or text alongside

## 6. Component Patterns

### Button States

Every button must implement all 5 states:

| State | Visual Treatment |
|-------|-----------------|
| Default | Base color, full opacity |
| Hover | Shift to `--color-interactive-hover`, cursor: pointer |
| Active / Pressed | Scale to 0.97–0.98, darken |
| Focus | 2–3px outline using `--color-interactive`, offset 2px |
| Disabled | 38–50% opacity, cursor: not-allowed, non-interactive |

```css
/* Button base pattern */
.btn {
  padding: var(--space-3) var(--space-6);
  font-size: var(--text-sm);
  font-weight: 500;
  border-radius: /* consistent with design system radius */;
  transition: all 150ms ease-out;
  cursor: pointer;
}
.btn:hover  { background: var(--color-interactive-hover); }
.btn:active { transform: scale(0.97); }
.btn:focus-visible { outline: 2px solid var(--color-interactive); outline-offset: 2px; }
.btn:disabled { opacity: 0.4; pointer-events: none; cursor: not-allowed; }
```

**Variants**: Primary (filled), Secondary (outlined), Ghost (text only), Destructive (error color)
**Rule**: One primary CTA per screen. Secondary actions visually subordinate.

### Card Structure

```
┌─────────────────────────────────┐
│  [optional: media / header]     │  ← full-bleed or padded
│  ─────────────────────────────  │
│  Title          [badge / label] │  ← space-4 horizontal padding
│  Subtitle / description         │  ← text-secondary
│  ─────────────────────────────  │
│  [action area: links / buttons] │  ← space-4 padding, border-top
└─────────────────────────────────┘
```

Rules:
- Consistent `border-radius` from design token
- Hover state: subtle elevation change or border-color shift — not layout change
- No layout shift on hover (don't move surrounding elements)
- Cards in a grid: same height via CSS Grid `align-items: stretch`

### Form Layouts

**Single column** (default for most forms):

```
[Label]              ← font-weight: 500, text-sm
[Input field]        ← height min 44px, full-width
[Helper text]        ← text-xs, text-secondary, always visible
[Error message]      ← text-xs, color-error, appears below field on blur
```

Rules:
- Always use visible `<label>` — never placeholder-only labels
- Validate on **blur**, not on keystroke
- Error message: state cause + how to fix (not just "Invalid")
- Required fields: mark with asterisk + legend
- Multi-step forms: show step indicator, allow back navigation
- Auto-save long forms to prevent data loss

## 7. Backgrounds & Visual Depth

Don't default to solid backgrounds. Create atmosphere:

- **Gradient mesh**: Radial gradients layered at low opacity
- **Noise texture**: Subtle grain overlay (opacity 3–8%) adds tactility
- **Geometric patterns**: SVG-based, aligned to grid
- **Layered transparencies**: Cards with `backdrop-filter: blur()` against textured bg
- **Dramatic shadows**: Large, soft, colored drop shadows — not gray box shadows
- **Decorative borders**: Gradient or accent-colored borders on key elements

Match depth treatment to aesthetic tone — minimalist designs use restraint; maximalist designs lean into these effects.

## 8. Motion & Animation

| Rule | Value |
|------|-------|
| Micro-interactions | 150–300ms |
| Page transitions | 250–400ms |
| Complex transitions | ≤ 400ms max |
| Easing in | `ease-in` |
| Easing out | `ease-out` (entering elements) |
| Spring feel | Physics-based curves preferred over cubic-bezier |
| Exit animations | 60–70% of enter duration |
| List stagger | 30–50ms per item |

Rules:
- Animate `transform` and `opacity` only — never `width`, `height`, `top`, `left`
- Page load: one well-orchestrated staggered reveal > scattered micro-interactions
- Every animation must express cause-effect — no decorative-only motion
- Always implement `prefers-reduced-motion` override
- Animations must be interruptible by user input

---

## 9. Accessibility Non-Negotiables

- Color contrast body text: **4.5:1 minimum**
- Color contrast large text / UI elements: **3:1 minimum**
- Visible focus rings on all interactive elements (never `outline: none` without replacement)
- Touch targets: **44×44px minimum**
- All icon-only buttons have `aria-label`
- Form errors use `aria-live` or `role="alert"` for screen readers
- Tab order matches visual order
- Color is never the only indicator of state

---

## 10. Pre-Delivery Checklist

Before delivering any UI component or page:

### Visual
- [ ] No emoji used as icons
- [ ] All sizes/spacing from tokens — no arbitrary px values
- [ ] No raw hex in component styles — semantic tokens only
- [ ] Font sizes from type scale only
- [ ] One primary CTA per screen

### Interaction
- [ ] All 5 button states implemented
- [ ] Touch targets ≥ 44px
- [ ] Hover/active/focus/disabled all visually distinct
- [ ] Animations use transform/opacity only
- [ ] `prefers-reduced-motion` respected

### Forms
- [ ] Visible labels on all inputs
- [ ] Error messages below field with cause + fix
- [ ] Validation on blur, not keystroke

### Quality
- [ ] Doesn't look like generic AI output
- [ ] Has a clear, committed aesthetic direction
- [ ] Tested at 375px mobile width
- [ ] Contrast checked in both light and dark mode
- [ ] No layout shift on hover/interaction
